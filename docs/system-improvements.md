# 系统改进文档

## 统一登录机制设计

根据需求，我们对系统进行了改进，使后台管理系统采用与主平台相同的用户登录方式。主要改进内容如下：

### 1. 统一用户登录入口

- 移除了专用的管理员登录控制器和服务方法
- 所有用户（管理员、教师、学生）通过统一的`/user/login`接口登录
- 登录成功后根据用户角色自动设置相应权限

### 2. 权限控制机制

- 使用SaToken框架实现基于角色的权限控制
- 在登录时自动设置用户角色
- 设置路由拦截器，根据用户角色控制接口访问权限：
  - `/admin/**` 路径仅允许管理员访问
  - `/teacher/**` 路径允许教师和管理员访问
  - `/student/**` 路径允许任何已登录用户访问

### 3. 服务层改进

- 移除了`AdminService`中的登录方法，统一使用`UserService`
- 改进了`UserService.login()`方法，登录时自动设置用户角色
- 保留了`AdminService`中的管理员特有功能，如邀请码管理

## 知识库权限模型

针对知识库系统，我们实现了精细的权限控制模型：

### 1. 用户角色与权限

- **管理员**：
  - 可以创建和管理自己的知识库
  - 可以查看系统中所有知识库，但只能编辑自己创建的知识库
  - 管理邀请码系统

- **教师**：
  - 可以创建和管理自己的知识库
  - 只能查看自己创建的或已加入的知识库
  - 可以上传教学资源

- **学生**：
  - 通过邀请码加入知识库
  - 只能查看已加入的知识库（只读权限）

### 2. 知识库操作权限

- **创建知识库**：管理员和教师可以创建
- **编辑知识库**：只有创建者可以编辑自己的知识库
- **删除知识库**：只有创建者可以删除自己的知识库
- **查看知识库**：
  - 管理员可以查看所有知识库
  - 教师只能查看自己创建或加入的知识库
  - 学生只能查看已加入的知识库

### 3. 知识点操作权限

- **添加/编辑/删除知识点**：只有知识库创建者可以操作
- **查看知识点**：所有有权限访问知识库的用户都可以查看

## 系统组件说明

### 1. 控制器层

- **UserController**：处理所有用户的登录、注册等通用操作
- **AdminController**：处理管理员特有的功能，如邀请码管理
- **AdminKnowledgeBaseController**：管理员的知识库操作
- **TeacherKnowledgeBaseController**：教师的知识库操作
- **StudentKnowledgeBaseController**：学生的知识库操作

### 2. 服务层

- **UserService**：用户通用服务，处理登录、注册等
- **UserRoleService**：用户角色服务，处理角色判断和权限设置
- **AdminService**：管理员特有功能服务，如邀请码管理
- **KnowledgeBaseService**：知识库核心服务
- **KnowledgeBaseServiceExtension**：知识库扩展服务，处理知识点操作

### 3. 配置类

- **SaTokenConfiguration**：SaToken配置类，设置路由拦截和权限控制规则
